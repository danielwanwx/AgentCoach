It is easy to get these terms mixed up because they all serve as "IDs," but they operate at different layers of the process.Here is the full breakdown of how a chunked upload works, using the specific terminology you asked about.1. Key Terminology DefinitionsUpload ID (The Session): Think of this as the "Ticket Number" for the entire job. It’s a unique string generated by the cloud provider when you first say, "I want to upload video.mp4." Every chunk you send must include this ID so the server knows which file the chunk belongs to.

Chunk/Part Number: Chunks are identified by a simple integer (1, 2, 3...). You must tell the server, "This is Part 5 of Upload ID XYZ."

ETag (The Receipt): When you upload a chunk, the cloud provider stores it and sends back an ETag (usually an MD5 hash of that specific chunk). It is the server's way of saying, "I received these specific bytes correctly." You must save these ETags to "finalize" the upload later.

Fingerprint (The Client-Side ID): This is not a cloud provider term; it’s a client-side trick (used by libraries like Uppy or TUS). It’s a hash of the file name, size, and last modified date. It allows the browser to remember: "Oh, I was halfway through uploading this exact file yesterday," even if the Upload ID was lost.2.

The Full Execution Flow

Phase 1: Initialization
Client calculates a Fingerprint for the local file (e.g., myvideo_100MB_Jan18).Client asks the App Server: "Do you have an active Upload ID for this Fingerprint?"If no, the App Server calls S3/GCS to get a new Upload ID and sends it to the client.

Phase 2: The Chunk LoopClient slices the file. It sends Part 1 + Upload ID.Cloud Provider stores the bytes and returns an ETag for Part 1.App Server (or Client) records: Part 1 = Success (ETag: abc123...).This repeats for Part 2, 3, etc.

Phase 3: Handling Failure (The Resume)
The connection drops at Part 20.The Client reconnects and sends the Fingerprint.The App Server looks up its database and sees Parts 1–19 are done.Crucial Step: To be 100% safe, the Server can call the Cloud's ListParts API using the Upload ID to verify the cloud actually has 1-19.The Server tells the Client: "I have everything up to Part 19. Start sending from Part 20.

"Phase 4: Finalization (The Assembly)Once the last part is uploaded, the Client sends a "Complete" request.This request must include a list of every Part Number and its corresponding ETag.The Cloud Provider looks at the list, stitches those specific chunks together in order, and creates the final file

Both are true! S3's multipart upload API combines chunks into a single object after upload completes, but Dropbox implements its own chunking layer on top of that for delta sync. DB splits files into 4MB chunks, hashes each chunk, and keeps its own mapping of these hashes to S3 objects. When a file changes, Dropbox's client identifies which chunks changed through hash comparison, then only uploads those chunks to S3.
